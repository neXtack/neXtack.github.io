    【Spring源码学习】0x03 SPI 机制

    vuePress-theme-reco iYoung    2021

    【Spring源码学习】0x03 SPI 机制

    iYoung 2021-5-5 Spring

    Spring环境搞好了，先不着急，还是先看一下这个SPI，Spring 中大量使用了SPI机制，所以先来学习下这个SPI。

    # 定义

    SPI，全称Service Provider Interface，是Java提供的一种发现和加载实现的一种机制。这个本来是用于Java内部的机制，API也是私有的，在Java 6的时候，将这种机制公开了出来。

    SPI包含以下几个部分：

    • 调用方
    • 服务接口
    • 服务实现
    • ServiceLoader

    # 调用方

    调用方很容易理解，就是使用这些服务的地方，要不然，服务没人用，要服务就没有意义了。

    # 服务接口

    定义一个接口，把功能抽象出来，这个部分既贴近调用方，供调用方使用，又贴近服务实现，约定实现。

    # 服务实现

    实现服务接口，完成具体的功能逻辑。另外还要在classpath下增加配置目录META/services/，添加一个文本文件，文件名为服务接口的完全限定名，内容就是服务实现的完全限定名。

    # ServiceLoader

    这个是Java提供的工具类，用于发现和加载服务，主要在调用方中实现，来加载具体的实现。

    # java.sql.Driver

    我们也不写demo了，直接看Java内部使用的一个示例——java.sql.Driver。

    先看一段代码：

    public class DriverDemo {

    public static void main(String[] args) throws Exception {
        String url = "jdbc:mysql://localhost:3306/demo?useUnicode=true&characterEncoding=utf-8&useSSL=false&serverTimezone=Asia/Shanghai&allowMultiQueries=true&allowPublicKeyRetrieval=true";
        String username = "root";
        String password = "123456";

        Connection connection = DriverManager.getConnection(url, username, password);
        System.out.println(connection.getMetaData().getDriverName());
    }
}

    这段代码，向我本地的mysql数据库建立连接，但我并没有添加mysql的驱动依赖。

    执行这段代码，抛出了异常：java.sql.SQLException: No suitable driver found for jdbc:mysql://localhost:3306...。

    添加依赖：

    <dependency>
    <groupId>mysql</groupId>
    <artifactId>mysql-connector-java</artifactId>
    <version>8.0.16</version>
</dependency>

    再次执行这段代码，就正常了，打印出MySQL Connector/J。

    很神奇，DriverManager可以自动获取到mysql驱动，就是借助了SPI机制。

    先把几个角色明确一下：

    • java.sql.DriverManager属于调用方
    • java.sql.Driver属于服务接口
    • com.mysql.cj.jdbc.Driver属于服务实现

    先看看DriverManager是怎么找到这个服务实现的。

    static {
    loadInitialDrivers();
    println("JDBC DriverManager initialized");
}

    DriverManager内部有个静态代码块，调用了本地静态方法的loadInitialDrivers。

    private static void loadInitialDrivers() {
    ServiceLoader<Driver> loadedDrivers = ServiceLoader.load(Driver.class);
    Iterator<Driver> driversIterator = loadedDrivers.iterator();

    try{
        while(driversIterator.hasNext()) {
            driversIterator.next();
        }
    } catch(Throwable t) {
    // Do nothing
    }
    return null;
}

    仅保留跟SPI相关的代码

    可以看到，这里使用ServiceLoader的静态方法load，得到一个ServiceLoader实例，然后就得到了Driver的迭代器。

    所以重点还是在ServiceLoader：

    public final class ServiceLoader<S> implements Iterable<S> {}

    ServiceLoader实现了Iterable接口。

    看静态方法load，最终到了这里：

    return new ServiceLoader<>(service, loader);

    继续看，然后到了reload这个实例方法这里：

    providers.clear();
lookupIterator = new LazyIterator(service, loader);

    这里就是对字段进行赋值，不过出现了一个LazyIterator。

    也就是说ServiceLoader.load(Driver.class)这个里面并没有做实质性的工作。

    那就看回loadedDrivers.iterator()这句，看ServiceLoader#iterator方法：

    public Iterator<S> iterator() {
    return new Iterator<S>() {

        Iterator<Map.Entry<String,S>> knownProviders
            = providers.entrySet().iterator();

        public boolean hasNext() {
            if (knownProviders.hasNext())
                return true;
            return lookupIterator.hasNext();
        }

        public S next() {
            if (knownProviders.hasNext())
                return knownProviders.next().getValue();
            return lookupIterator.next();
        }

        public void remove() {
            throw new UnsupportedOperationException();
        }

    };
}

    返回了一个新的迭代器，这里也没有寻找服务的代码。

    那就再看回代码：

    while(driversIterator.hasNext()) {
	driversIterator.next();
}

    就是遍历这个loadedDrivers.iterator()的迭代器，两个方法hasNext和next。 简化一下：

    public boolean hasNext() {
	return lookupIterator.hasNext();
}
public S next() {
	return lookupIterator.next();
}

    看来重点应该在lookupIterator，也就是LazyIterator中了。 这两个方法对应起来就是：

    private boolean hasNextService() {
    String fullName = PREFIX + service.getName();
    if(configs == null) {
        configs = loader.getResources(fullName);
    }
    pending = parse(service, configs.nextElement());
}

private S nextService() {
    String cn = nextName;
    nextName = null;
    Class<?> c = Class.forName(cn, false, loader);
    if (!service.isAssignableFrom(c)) {
        fail(service,
                "Provider " + cn  + " not a subtype");
    }
    S p = service.cast(c.newInstance());
    providers.put(cn, p);
}

    PREFIX是个常量：META-INF/services/，service就是java.sql.Driver.class，所以fullName就是META-INF/services/java.sql.Driver。 然后调用ClassLoader#getResources方法，返回一个Enumeration<URL>，就是获取classpath下所有的这个文件，然后进行解析。

    parse(service, configs.nextElement())内部的逻辑，就是解析文本文件的每一行，然后返回，也就是说一行一个实现，一个实现一行，可以多行。

    通过hasNextService就获取到了服务实现的类名。

    这里还没有获取到我们代码中能使用的Driver实例对象，那就再看nextService：

    Class<?> c = Class.forName(cn, false, loader);
S p = service.cast(c.newInstance());

    到这里算是结束了。

    回头看一下：

    private static void loadInitialDrivers() {
    ServiceLoader<Driver> loadedDrivers = ServiceLoader.load(Driver.class);
    Iterator<Driver> driversIterator = loadedDrivers.iterator();

    try{
        while(driversIterator.hasNext()) {
            driversIterator.next();
        }
    } catch(Throwable t) {
    // Do nothing
    }
    return null;
}

    这里为什么要‘空转’，原来driversIterator.next()在这一步才完成服务实现类的初始化，到这里才算完成了服务发现和加载的过程。

    # 优缺点

    总结一下，SPI的优点就是解耦，定义与实现相分离，调用方可以根据实际情况来决定使用哪个实现。

    但是，缺点也很明显：

    1. ServiceLoader是final的，也没有监听器，这么一个核心的类，毫无扩展点。
    2. ServiceLoader线程不安全（这个也不算缺点吧，由调用方去控制线程安全问题也是可取的）。
    3. ServiceLoader获取实现时，虽然是懒加载，但意义不大，无法获取特定实现。

    然后看下约束，要使用SPI有哪些限制：

    1. 服务实现要提供一个无参构造器。
    2. 服务实现要在META/services声明自己
    3. 调用方要处理并发问题
    4. 调用方要有处理多个实现的逻辑

    # Spring扩展

    Spring 也大量使用SPI机制，但是Spring采用的是自己设定的一套SPI机制。

    主要看Spring的三个文件：

    • META-INF/spring.factories
    • META-INF/spring.handlers
    • META-INF/spring.schemas

    与Java SPI有哪些不同：

    1. 声明文件命名方式不同，Spring根据作用进行分类，不再使用接口名作为文件名称
    2. 声明文件内容不同，Spring采用key=value的形式，而且value可以以,分割
    3. 加载工具类不同，Spring不再使用ServiceLoader来加载
    4. 服务实现类可以不同暴露无参构造器（即public），但是必须存在无参构造器

    那就看看META-INF/spring.factories的加载过程——org.springframework.core.io.support.SpringFactoriesLoader。

    它就只有两个静态公共方法：

    public static List<String> loadFactoryNames(Class<?> factoryClass, @Nullable ClassLoader classLoader) {
    String factoryClassName = factoryClass.getName();
    return loadSpringFactories(classLoader).getOrDefault(factoryClassName, Collections.emptyList());
}

    可以看到调用了loadSpringFactories方法，而且这个方法没有接收factoryClass参数信息，也就是说loadSpringFactories获取到了所有的spring.factories信息。 这也是没有办法，Spring把所有的spring.factories的声明都放在一起了，只能一起去解析。

    继续看loadSpringFactories方法：

    private static Map<String, List<String>> loadSpringFactories(@Nullable ClassLoader classLoader) {
    MultiValueMap<String, String> result = cache.get(classLoader);
    if (result != null) {
        return result;
    }

    try {
        Enumeration<URL> urls = (classLoader != null ?
                classLoader.getResources(FACTORIES_RESOURCE_LOCATION) :
                ClassLoader.getSystemResources(FACTORIES_RESOURCE_LOCATION));
        result = new LinkedMultiValueMap<>();
        while (urls.hasMoreElements()) {
            URL url = urls.nextElement();
            UrlResource resource = new UrlResource(url);
            Properties properties = PropertiesLoaderUtils.loadProperties(resource);
            for (Map.Entry<?, ?> entry : properties.entrySet()) {
                String factoryClassName = ((String) entry.getKey()).trim();
                for (String factoryName : StringUtils.commaDelimitedListToStringArray((String) entry.getValue())) {
                    result.add(factoryClassName, factoryName.trim());
                }
            }
        }
        cache.put(classLoader, result);
        return result;
    }
    catch (IOException ex) {
        throw new IllegalArgumentException("Unable to load factories from location [" +
                FACTORIES_RESOURCE_LOCATION + "]", ex);
    }
}

    首先，结果是MultiValueMap类型，也就是一个key，可以对应多个value。

    流程跟ServiceLoader逻辑差不多，不过简化了好多，直接调用ClassLaoder#getResources方法，加载到所有META-INF/spring.factories，由于Spring SPI采用key=value的形式，直接使用了Properties来保存解析结果，然后进一步处理。 由于value是可以以,分割的，对value要做一次分割处理。

    然后看loadFactories方法：

    public static <T> List<T> loadFactories(Class<T> factoryClass, @Nullable ClassLoader classLoader) {
    List<String> factoryNames = loadFactoryNames(factoryClass, classLoaderToUse);
    List<T> result = new ArrayList<>(factoryNames.size());
    for (String factoryName : factoryNames) {
        result.add(instantiateFactory(factoryName, factoryClass, classLoaderToUse));
    }
    AnnotationAwareOrderComparator.sort(result);
    return result;
}

    这里调用loadFactoryNames获取到factoryClass所有实现类的名称，然后调用instantiateFactory实例化:

    private static <T> T instantiateFactory(String instanceClassName, Class<T> factoryClass, ClassLoader classLoader) {
    Class<?> instanceClass = ClassUtils.forName(instanceClassName, classLoader);
    return (T) ReflectionUtils.accessibleConstructor(instanceClass).newInstance();
}

    还是用到了反射，与ServiceLoader一样，但是调用newInstance时，去获取了无参构造器，并且调用了setAccessible(true)。

    到这里，服务实现就全部加载到了，但是Spring还多了一个步骤：AnnotationAwareOrderComparator.sort(result)，对所有实例进行排序，排序依据就是Spring的bean排序机制。

    Spring的SPI机制，简化了处理逻辑，增加了服务实现的优先级策略，但是不具备通用性，是在Spring体系下的一种SPI机制。

    # 总结

    SPI机制本身就是为了解耦，尤其是对于一些标准规范类的API，可以供第三方提供自己的实现。无论从原理上还是实现上来说，SPI机制还是比较简单的，如果需要的话，也可以对SPI进行必要的扩展，从而更加适应自己的框架或者项目。
